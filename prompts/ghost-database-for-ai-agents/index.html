The prompt

Design a persistent memory and knowledge layer — a 'ghost database' — for a long-running AI agent system built with [framework/stack]. The system must support: short-term episodic memory (recent conversation turns), long-term semantic memory (vector search over past interactions and documents), structured fact storage (key-value entities the agent can update), and a retrieval strategy that selects the right memory type at inference time. Provide the full schema, the retrieval orchestration logic, a sample system-prompt injection template, and guidance on when to write vs. read from each store.

Sample output

**Ghost Database Architecture** **Memory layers:** | Layer | Store | TTL | Use case | |---|---|---|---| | Episodic (short-term) | Redis list (sliding window) | Last 20 turns | Recent dialogue context | | Semantic (long-term) | ChromaDB / Pinecone | Permanent | Past interactions, docs | | Structured facts | PostgreSQL key-value table | Permanent, mutable | User prefs, entity state | **Structured facts schema:** ```sql CREATE TABLE agent_facts ( agent_id TEXT, key TEXT, value JSONB, confidence REAL, updated_at TIMESTAMPTZ, PRIMARY KEY (agent_id, key) ); ``` **Retrieval orchestration (pseudocode):** ```python def build_context(query, agent_id): episodic = redis.lrange(f'ep:{agent_id}', 0, 19) # always include semantic = chroma.query(query, n_results=4) # top-4 semantic hits facts = pg.fetch('SELECT key, value FROM agent_facts WHERE agent_id=$1', agent_id) return format_context(episodic, semantic, facts) ``` **System-prompt injection template:** ``` [MEMORY] Recent context: {episodic_summary} Relevant past knowledge: {semantic_hits} Known facts about user: {facts_json} [/MEMORY] ``` **Write rules:** - After every turn → append to episodic Redis list, pop oldest if > 20 - After task completion → embed summary → upsert to vector DB - When agent learns a stable fact (name, preference, goal) → upsert to PostgreSQL facts table
